# Capability 11: Application / Solution Architecture Modelling

**Domain:** Architecture Development – Application Architecture

**Tagline:** Design and document application structures to guide development and transformation.

## Definition

Application Architecture Modelling involves designing and documenting the structure, components, and interactions of software applications. It provides a blueprint for how applications are composed, relate to each other, and support business capabilities.

## Objectives

Ensure that applications are designed and implemented in a way that aligns with business goals and IT strategy.
- Facilitate effective communication and collaboration among stakeholders.
- Support decision-making and prioritization of application development and integration initiatives.

## Tasks

1. **Analyse the current application landscape** — document applications, technology stack and lifecycle status.
2. **Define application architecture principles,** standards and design patterns (e.g. microservices, layered, event-driven).
3. **Design target application architectures** — model components, responsibilities, interactions and data flows.
4. **Identify integration points** and dependencies with other architecture domains.
5. **Validate application models** with stakeholders and iterate based on feedback.
6. **Maintain and update** application models as requirements and the technology landscape evolve.

## Inputs

- Business strategy and objectives
- Current state application documentation
- Stakeholder requirements and expectations
- Architecture Vision and principles
- Application portfolio inventory

## Outputs

- Application architecture models (e.g. component diagrams, data flow diagrams)
- Application architecture principles and standards
- Integration points and interfaces documentation
- Architecture Decision Records (ADRs)

## Roles

- Application Architect (R)
- Lead Enterprise Architect (A)
- Enterprise Architect, Solution Architect (C)
- Business Analyst, Technology Architect (I)

## Common Issues with an Immature Capability

- No standardized way of documenting application architectures — teams use different tools and notations.
- Architecture only done at project level — no enterprise-wide view, leading to silos and duplication.
- Architecture decisions are not documented, leading to repeated debates and inconsistent choices.

## Pragmatic Approach – Essentials That Are Needed for Success

- Start with an application portfolio inventory — you can't design what you don't understand.
- Define a standard notation and toolset (e.g. ArchiMate, C4 model).
- Focus on business-critical applications first — not everything needs a detailed model.
- Connect application architecture to business capabilities — this creates relevance and stakeholder buy-in.
