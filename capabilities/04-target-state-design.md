# Capability 4: Target State Design

**Domain:** Architecture Development - Common

## Purpose

To define the desired future-state ("to-be") architecture that is aligned with business strategy, goals, and guiding principles, translating strategic intent into concrete architecture blueprints across all domains.

## Scope

- Architecture vision and future state definition
- Blueprint creation across business, application, data, and technology domains
- Architecture option analysis and trade-off evaluation
- Transition architecture definition (intermediate states)

## Key Activities

- Develop architecture vision aligned with business strategy
- Design target state architecture across all domains
- Apply architecture principles and standards as design constraints
- Perform option analysis and trade-off evaluation (e.g., ATAM)
- Define transition architectures (intermediate states between current and target)
- Create architecture blueprints and models
- Validate target state with stakeholders
- Document architecture decisions and rationale (ADRs)
- Assess feasibility and impact of the target state

## Inputs

- Business strategy and goals
- Current state assessment and baseline
- Architecture requirements (functional and non-functional)
- Architecture principles and standards (Cap 6)
- Industry trends and best practices
- Constraints (budget, timeline, technology, regulatory)

## Outputs / Deliverables

- Target state architecture models and blueprints
- Architecture vision document
- Option analysis and trade-off documentation
- Architecture decision records (ADRs)
- Transition architecture definitions
- Gap analysis inputs (current vs. target)

## Roles & Responsibilities

| Role | Responsibility |
|------|---------------|
| Chief Architect | Sets architecture vision and strategic direction |
| Enterprise Architect | Designs enterprise-wide target state |
| Domain Architects | Design domain-specific target states |
| Solution Architects | Validate feasibility from a solution perspective |

## Maturity Levels

| Level | Name | Description |
|-------|------|-------------|
| 1 | Initial | No formal target state; changes are reactive and project-driven |
| 2 | Repeatable | Target state defined for individual projects or domains |
| 3 | Defined | Enterprise-wide target state defined and aligned with strategy; blueprints exist |
| 4 | Managed | Target state regularly reviewed and updated; actively used to guide investment decisions |
| 5 | Optimizing | Adaptive target state continuously evolving with strategy; scenario planning and what-if analysis used |

## Key Metrics / KPIs

- Strategic alignment score of target architecture
- Percentage of initiatives aligned with target state
- Number of architecture decisions documented (ADRs)
- Stakeholder approval rate for target designs
- Gap closure rate over time

## Dependencies

- **Current State Assessment** (Cap 3) - Gap between current and target drives design
- **Requirements Management** (Cap 2) - Requirements constrain and shape the target
- **Gap Analysis & Roadmapping** (Cap 5) - Target state is the destination for the roadmap
- **Architecture Principles & Standards** (Cap 6) - Principles and standards guide target design
- **Architecture Decision Management** (Cap 27) - Decisions made during target state design are governed

## Tools & Technologies

- Architecture modelling tools (Archi, Sparx EA, LeanIX, Mega)
- Scenario planning and what-if analysis tools
- Trade-off analysis techniques (ATAM, CBAM)
- Reference architecture libraries
- Collaboration and whiteboarding tools
