# Capability 16: System Architecture Modelling

**Domain:** Architecture Development – Technology Architecture

**Tagline:** Define and design the technical systems and infrastructure components.

## Definition

System Architecture Modelling involves designing and documenting the technical infrastructure and system components that host and run business applications. It defines servers, compute resources, operating systems, middleware, databases, and their configurations.

## Objectives

Design robust and scalable system architectures that support application requirements while optimizing performance, availability, security, and cost. Ensure consistent standards and patterns across the technical infrastructure. Enable effective capacity planning, disaster recovery, and operational management.

## Tasks

1. **Identify system requirements** from application architecture, business requirements, and non-functional requirements.
2. **Define system components, topology and configuration** for servers, compute, operating systems, middleware, data platforms, and storage – their deployment, connection, and configuration.
3. **Apply system architecture patterns and standards** (e.g., high availability, load balancing, clustering, redundancy).
4. **Define sizing and capacity requirements** based on workload analysis and growth projections.
5. **Document security & compliance** and validate system architecture with stakeholders (application architects, infrastructure teams, security).
6. **Maintain and evolve** system architecture as technology and business requirements change.

## Inputs

- Application architecture and requirements
- Non-functional requirements
- Technology standards and reference architectures
- Existing infrastructure, capacity and performance data
- Budget and cost constraints

## Outputs

- System architecture diagrams and models
- System component, capacity and sizing specifications
- Deployment architecture documentation
- High availability and disaster recovery designs
- Security and compliance requirements

## Roles

- Technology Architect (A)
- Application Architect
- System Administrator
- Lead Enterprise Architect

## Common Issues with an Immature Capability

- Inconsistent system designs across different applications and environments.
- Over- or under-provisioning due to lack of proper sizing and capacity planning.
- Lack of standardization in system platforms, configurations, and patterns.
- Inadequate disaster recovery planning.
- Security gaps due to inconsistent hardening and configuration standards.
- High operational costs from inefficient infrastructure design and complexity.

## Pragmatic Approach – Essentials That Are Needed for Success

- Start with non-functional requirements: performance, availability, scalability.
- Use reference architectures and proven patterns – leverage cloud-first.
- Balance cost, performance, and resilience - optimize for business value.
- Standardize - reduce complexity and operational overhead.
- Design for observability - build monitoring, logging & diagnostics from the start.
- Design for automation and ensure operational feasibility.
- Document - include diagrams, specifications, and rationale for design decisions.
